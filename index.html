Market Problem

招聘市场不缺简历和岗位，缺的是可信的能力数据。

传统招聘平台用简历、JD、关键词和沟通效率解决问题，但真正影响成败的是候选人真实能力、岗位真实要求和组织真实环境。

下一代招聘和职业发展平台的核心资产，不再是简历数量，而是真实能力数据。

Echo 的底层判断
候选人侧简历会被包装
企业真正想知道他承担了什么角色，如何解决问题，压力下如何决策。
Echo 的方式从真实职场故事里抽取行为证据和能力信号。
企业侧JD 往往抽象
候选人真正想知道岗位要解决什么业务问题，组织环境和上级风格如何。
Echo 的方式把岗位转化为结构化岗位能力矩阵。
平台侧匹配停在浅层
真实瓶颈关键词匹配无法判断 0.5 档、组织适配和能力证据。
Echo 的方式用能力星图和岗位矩阵进行可解释匹配。

Product Solution

前台是陪伴，后台是能力还原。

Echo 不让用户一上来上传简历或做测评，而是从“最近工作还顺吗”开始，让用户在自然交流中讲出真实事件。

Echo 的完整链路是：低防备职场对话 → 真实职场故事 → 行为证据抽取 → 个人能力星图 → 企业岗位能力矩阵 → 匿名授权匹配 → 猎头服务闭环。

C 端体验

像懂职场的旁观者朋友

陪用户吐槽、复盘、判断下一跳、准备面试，把混乱的职场问题说清楚，但不急着教育用户。

后台能力

把自然语言转成能力证据

识别背景、角色、冲突、行动、结果和情绪，再映射成能力标签、阶段判断和证据链。

MVP1 Focus

MVP1 不验证招聘平台，先验证“它真的懂我”。

第一阶段最重要的问题是：用户愿不愿意对 Echo 讲真实职场故事，并且在看到反馈后产生共鸣。

低防备职场对话 + 真实职场事件还原 + 职场发现卡

MVP1 的产品本质不是聊天，也不是招聘撮合，而是跑通“讲真话 → 被理解 → 愿意继续聊”的最小闭环。

2Echo 情绪承接，并每次只轻追问一个问题
3系统还原背景、角色、冲突、行动、结果和感受
4生成职场发现卡，用户反馈准不准

Technology

Echo 是多模型、多 Agent、多知识库的数据闭环系统。

技术系统不是把所有任务丢给一个大模型，而是让 Agent、模型、知识库、能力图谱和合规系统分工协作。

九层技术架构

1客户端层：Web / H5 / B 端后台 / 猎头工作台
2API 网关层：鉴权、限流、路由、日志、成本统计
3Agent 编排层：Orchestrator、Companion、Mentor、Matching
4模型服务层：LLM、抽取、分类、匹配、安全模型
5RAG 知识库层：职场、导师、猎头经验、岗位库
6能力图谱层：能力本体、岗位能力、职级能力、0.5 档模型
7业务数据层：用户、对话、事件、证据、星图、岗位、匹配
8安全合规层：授权、脱敏、审计、风险拦截
9运营评估层：标注后台、模型评估、用户运营、客户成功

Conversation Scenes

场景库不是互斥的树，而是识别真实表达的后台语言。

前台不需要给用户设置入口。用户只要自然说话，后台识别一个主场景、多个辅助标签、用户意图、情绪状态和当前对话阶段。

Roadmap

从 C 端陪伴对话 MVP，走向能力通行平台。

Echo 的推进顺序应克制：先验证用户愿意讲真话，再做能力星图与 0.5 档识别，最后连接 B 端岗位矩阵和商业化。

Phase 1

C 端陪伴对话 MVP

验证真实故事提交率、平均对话轮次、发现卡满意度和 7 日留存。

Phase 2

能力星图与 0.5 档识别

从有趣陪伴升级为可信能力还原系统，加入能力证据和深度报告。

Phase 3

B 端岗位能力矩阵接入

连接云略现有企业客户，建立第一批岗位能力矩阵和匿名匹配。

Phase 4

C/B 双边商业化

形成 C 端会员、按次付费、B 端席位和轻猎头服务收入。

Phase 5

能力通行平台化

沉淀行业能力标准、企业能力矩阵库、职业成长路径和双向信用体系。
